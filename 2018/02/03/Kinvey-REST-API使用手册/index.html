Kinvey - REST API使用手册

发表于 2018-02-03 | 分类于 BaaS |

Data Store/数据存储

Kinvey最简单的用例就是将数据存储到云端后端并从中取回数据.
数据的基本单位是一个实体,同一类的实体被组织在集合中. 实体是以JSON格式存储在后端的一组键值对. Kinvey的库自动将您的本机对象转换为JSON.
Kinvey的数据存储提供简单的数据CRUD操作,以及强大的过滤和聚合功能.
Appdata API构成了Kinvey后端提供的核心REST服务之一.

Entities/实体

Entity是数据的基本单位,在后端使用JSON表示形式存储为数据对象. 全部的CRUD操作可用于操纵 Entity.

PUT / appdata /:appKey /:collectionName /:id 充当upsert. 如果请求包含一个现有的实体ID,它会更新实体. 如果ID未被识别,请求会创建一个新的实体.目前没有办法只传递你想改变的属性. 后端将整个JSON正文存储为在请求正文中传递.

Delete

删除位于指定URI处的明确标识的实体或使用查询过滤的集合中的多个实体.使用API版本1或更低版本时,删除操作会返回“204无内容”响应.

从API版本2开始,删除将返回一个“200 OK”,其中包含在响应正文中删除的条目数量.

Required headers/参数必选项

  • Authorization

Deleting a collection

要永久删除整个集合,请将包含要删除集合名称的POST请求发送到/rpc/:appKey/remove-collection. 为确保这种破坏性行为不会意外发生,您必须使用主密钥(master secret)进行身份验证,并将X-Kinvey-Delete-Entire-Collection标头作为请求的一部分.

如果收集成功删除,Kinvey将以“200 OK”回应,其中包含删除收集数量的实体,总是1.

Required headers/参数必选项

  • Authorization
  • X-Kinvey-Delete-Entire-Collection: true
  • Content-Type: application/json

Metadata

每个实体都有一个与之关联的_kmd(Kinvey Metadata)JSON对象,它提供关于一个entity的 meta-information. 对象的格式是:

  • lmt(上次修改时间) - 保存实体上次在服务器上更新的时间.
  • ect(实体创建时间) - 保存实体在服务器上创建的时间. 当使用POST或PUT创建实体时,该属性将被添加到实体中,并且在实体的生命周期中将保持不变.

Restrictions and limitations/限制和限制

Kinvey对查询执行了一些限制.

####Regular expressions/常用表达

正则表达式需要锚定(以^为前缀),并区分大小写. 要执行不区分大小写的搜索,请在集合中创建一个规范化(即全部小写)的字段,然后在该字段上执行匹配.

####Disallowed operators/不允许的操作

$where和$query操作符不受支持,并且使用它们将导致错误.

####Result set limits/结果集限制

  • 提交给REST API的任何查询将返回最多10,000个结果. 如果您的查询匹配超过10,000个实体,则只返回前10,000个.
  • 一个查询只能检索100MB的数据,如果超过这个限制,将会返回一个错误. 检索大量的数据会给应用程序带来沉重负担,如果达到这个限制,我们鼓励您重新考虑您的设计. 如果您的应用程序绝对需要检索超过100MB的数据,请考虑使用分页.

Aggregation/聚合

要执行SQL样式的GROUP BY aggregation,请对_group进行POST. 请求主体应该包含一个具有四个属性的JSON对象:

  • key:选择要分组的字段的对象
  • initial:包含要返回的文档的初始结构的对象
  • reduce:将用于减少结果集的JavaScript函数
  • condition:在将结果集提供给MapReduce操作之前,应用于结果集的可选过滤器
