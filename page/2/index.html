基本介绍

SfChart基于MPAndroidChart,主要用于绘制各种图表,是先有了Android版本后才开发的iOS版本,两版本的API设计方式都是相同的,且以Android为准.因此SfCharts中组件代码架构可以根据MPAndroidChart架构来映射.主要分为LineData、Axis、MakerView、ViewPort.

先看一个基本图表所包含的元素.

  • 黄色区域:YAxis 负责Y轴的样式,如:刻度间距,显示刻度数量,背景横线,显示横向LimitLine
  • 绿色区域:XAxis 负责x轴的样式,如:刻度间距,显示刻度数量,背景横线,显示直向LimitLine
  • 紫色区域:Legend 负责图例的样式,如 图例形状 行距 间距 自定义颜色
  • 黑色箭头:LineData 图表数据与一些样式设定, 如 数据集合 线条样式 数据点样式

  • 橘色圆形:MakerView 点击数据点会弹出个Maker

    其中最重要的就是LineData,如果图表没有LineData数据,就只能单纯显示(No Chart data available)的文字,所以选择好图表类型并吧Chart放入Layout 第一步是将数据放入Chart中.

About App Development with UIKit

发表于 2017-06-08 | 分类于 iOS

官方文档链接

Overview 概述

UIKit框架提供了为iOS和tvOS构建应用程序所需的核心对象. 您可以使用这些对象在屏幕上显示您的内容,与该内容进行交互,并管理与系统的交互. 应用程序依赖于UIKit的基本行为,UIKit提供了许多方法来定制该行为以满足您的特定需求.

Xcode提供模板项目作为您创建的每个应用程序的起点. 例如,图1显示了使用Xcode中的单一视图应用程序模板创建的应用程序的结构. 模板项目提供最小的用户界面,因此您可以立即构建和运行项目,并在设备或模拟器中查看结果.

Figure 1
The structure of a single view iOS app

当您构建应用程序时,Xcode会编译您的源文件并为您的项目创建一个应用程序包. 应用程序包是一个结构化的目录,包含与应用程序关联的代码和资源. 资源包括支持代码的image assets,storyboard文件,strings文件和app metadata. 应用程序包的结构很重要,但Xcode知道您的资源需要去哪里,所以不要担心.

Required Resources

每个UIKit应用程序都需要具有以下资源:

  • App icons
  • Launch screen storyboard 启动屏 storyboard

系统会在主屏幕的“设置”中以及任何需要将应用程序与其他应用程序区分开来的位置显示您的应用程序图标. 由于它在多个地方以及多个设备上使用,因此您可以在Xcode项目的AppIcon图像资源中提供多个版本的应用程序图标,如图2所示.您的应用程序图标应该与众不同,以帮助用户快速识别您的应用程序 主屏幕. 但是,您可能会改变图标的细节以适应您必须提供的不同图像大小.

Figure 2
Providing icons for an iOS app

LaunchScreen.storyboard文件包含您的应用程序的初始用户界面,它可以是启动画面或实际界面的简化版本. 当用户点击你的应用程序的图标时,系统立即显示你的启动屏幕,让你的应用程序正在启动的用户. 启动屏幕还为自己初始化时的应用提供封面. 当您的应用程序准备就绪后,系统将隐藏启动屏幕并显示您的应用程序的实际界面.

Required App Metadata

系统从您的应用程序包中的信息属性列表（Info.plist）文件中导出关于您的应用程序的配置和功能的信息. Xcode为每个新的项目模板提供了这个文件的预配置版本,但是您可能需要在某个时候修改这个文件. 例如,如果您的应用程序依赖于特定的硬件或使用特定的系统框架,则可能需要将与这些功能相关的信息添加到此文件.

您可以对Info.plist文件进行的一项常见修改是声明您的应用程序的硬件和软件要求. 这些要求是您如何向系统传达您的应用程序需要运行的方式. 例如,导航应用程序可能需要GPS硬件的存在才能提供转由转向的指示,如图3所示.App Store阻止将应用程序安装在不符合应用程序要求的设备上.
Figure 3
Declaring your app’s hardware and software requirements

有关可以包含在Info.plist文件中的密钥的信息,请参阅 Information Property List Key Reference

Code Structure of a UIKit App

UIKit提供了许多应用程序的核心对象,包括与系统交互的应用程序,运行应用程序的主事件循环,并在屏幕上显示您的内容. 您可以按原样使用大部分对象,也可以仅使用较小的修改. 知道要修改哪些对象以及何时修改它们对于实现您的应用程序至关重要.

UIKit应用程序的结构基于模型 - 视图 - 控制器（MVC）设计模式,其中对象根据其目的进行划分. 模型对象管理应用程序的数据和业务逻辑. 查看对象提供了数据的可视化表示. 控制器对象充当模型和视图对象之间的桥梁,在适当的时候在它们之间移动数据.

图4代表了UIKit应用程序的一个相当典型的结构. 您提供表示您的应用程序的数据结构的模型对象. UIKit提供了大部分视图对象,尽管您可以根据需要为数据定义自定义视图. 协调您的数据对象和UIKit视图之间的数据交换是您的视图控制器和应用程序委托对象.

Figure 4
The core app objects

UIKit和Foundation框架提供了许多用于定义应用模型对象的基本类型. UIKit提供了一个UIDocument对象来组织属于基于磁盘文件的数据结构. Foundation框架定义了代表字符串,数字,数组和其他数据类型的基本对象. The Swift Standard Library提供了许多基础框架中可用的相同类型.

UIKit提供了应用程序的控制器和视图图层中的大部分对象. 具体来说,UIKit定义了UIView类,它通常负责在屏幕上显示你的内容. （您也可以使用Metal和其他系统框架直接在屏幕上显示内容.）UIApplication对象运行应用程序的主事件循环,并管理应用程序的整个生命周期.

Managing Your App's Life Cycle

发表于 2017-06-06 | 分类于 iOS

官方文档链接

Overview

UIKit应用程序总是处于五种状态之一.如图1所示.

  • 应用程序开始不运行(not runing).
  • 当用户显式启动应用程序时,应用程序在进入活动状态(Active)之前会短暂地移至非活动状态(inActive).(一个活跃的应用程序出现在屏幕上,被称为前台应用程序.)
  • 退出一个活动的应用程序会将其移出屏幕,并进入后台状态(Background),直到系统暂时停止(Suspend).
  • 系统可以自行决定终止一个被暂停的应用程序,并将其返回到非运行状态(not runing).

Figure 1
The execution states of a UIKit app

您的应用程序的当前状态定义了可用的系统资源.由于活动应用程序在屏幕上可见并且必须响应用户交互,因此在使用系统资源时它们具有优先权. 后台应用程序在屏幕上不可见,因此对系统资源的访问更加有限,并且执行时间有限.

Manage Life Cycle Events 管理生命周期事件

当您的应用程序从一种状态转换到另一种状态时,UIKit会通知您的应用程序委托对象-符合UIApplicationDelegate协议的对象.
使用您的应用程序委托来修改您的应用程序的行为以匹配其新的状态.例如,从非运行状态(not run)转移到非活动状态(inActive)时,处理准备运行应用程序所需的启动时任务.

系统通知您的应用程序delegate有关以下转换：

  • Launch. 您的应用程序从未运行状态转换为非活动状态或后台状态.使用此转换来准备您的应用程序运行;see Responding to the Launch of Your App.
  • Activation. 您的应用程序从非活动状态转换为活动状态.准备您的应用程序在前台运行,并在屏幕上可见; see Preparing Your App to Run in the Foreground.
  • Deactivation. 您的应用程序从活动状态转换到非活动状态.安静你的应用程序,也许只是暂时的;see Preparing Your App to Run in the Background.
  • Background execution. 您的应用程序将从非活动状态转换到后台状态. 准备在屏幕外运行时只处理必要的任务;see Preparing Your App to Run in the Background.
  • Termination. 您的应用从任何运行状态转换到未运行状态.(暂停的应用程序在终止时不会收到通知.)取消所有任务并准备退出; see applicationWillTerminate:.

Manage Behavioral Events 管理行为事件

您的 app delegate 也会响应其他一些重要事件:

  • Memory warnings. 减少您的应用程序使用的内存量;see Responding to Memory Warnings.
  • Time changes. Update time-sensitive features of your app.
  • Protected data becomes available/unavailable. 在用户锁定或解锁设备时管理文件.
  • State restoration. 将应用的用户界面恢复到之前的状态,使您的应用永不停止运行;see Preserving Your App’s UI Across Launches.
  • Handoff tasks. 继续在另一台设备上启动任务.
  • Open URLs. 接收并打开发送到您的应用的网址.
  • Inter-app communication. 从配对的watchOS应用程序接收数据.
  • File downloads. 接收使用您的应用下载的文件 NSURLSession object.

尽管应用程序委托是处理生命周期事件的主要场所,但并不是唯一的地方. 对于大多数事件,UIKit还会生成任何对象都可以观察的通知. 有关您可以观察的与应用程序相关的通知列表,see UIApplication.有关用于处理事件的方法的更多信息, see UIApplicationDelegate.

Apple 编程核心思想

发表于 2017-05-06 | 分类于 iOS

面向对象与面向过程

  • OC语言是面向对象,C语言是面相过程.它们是解决问题的两种思想.
  • 两者的不同点是:面向过程管住的是解决问题需要那些步骤,面向对象关注的是解决问题需要那些对象.
  • 面向对象的核心是对象,面向对象有两个特别重要的概念:类和对象.
  • 编程语言的对象建立,是通过类来创建.就相当于盖房子用的设计图纸,类就是创建对象的描述.对象是类的具体存在,一般在解决问题中都会先设计类,再通过类创建相应的对象.
  • 类的创建有三个属性:类名、属性、行为(功能),一般名词就是类名,切首字母必须大写,不能使用下划线.

面向协议编程(Protocol Oriented Programming,以下简称 POP)是 Apple 在2015年WWDC上提出的Swift的一种编程范式.
相比与传统的面向对象编程(OOP),POP 显得更加灵活.
结合 Swift 的值语义特性和 Swift 标准库的实现,这一年来大家发现了很多 POP 的应用场景.

Swift 标准库中有 50 多个复杂不一的协议,几乎所有的实际类型都是满足若干协议的.protocol 是 Swift 语言的底座,语言的其他部分正是在这个底座上组织和建立起来的.这和我们熟知的面向对象的构建方式很不一样.
