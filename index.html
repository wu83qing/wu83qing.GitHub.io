SiriKit Part 1: Hey Siri, How Do I Get Started?

发表于 2018-07-30 | 分类于 iOS

Siri是Apple的智能个人助理,它允许使用语音与iOS,watchOS,tvOS和macOS设备进行交互.已经被数百万人使用,这项技术将继续增长;事实上,MarketsAndMarkets预测到2023年语音和语音识别市场的价值将达到 达到183亿美元.为了继续在这个市场中茁壮成长,Apple提供了SiriKit,使开发人员可以更轻松的与他们的apps集成.

本系列将探讨SiriKit,为什么它很重要以及如何使用它来通过Siri公开你的app’s功能.我们将介绍SiriKit的基础知识以及如何为iOS app添加支持,深入了解Intent,然后解决最终的问题,以获得良好的Siri用户体验.

什么是SiriKit？

Siri通过提供对话界面而不是生硬的命令来与Siri交谈,最终目的是努力让用户感到自然;虽然这使得用户方便使用Siri,但对于开发人员来说这可能非常复杂.幸运的是,Apple通过处理对话方面来缓解这种复杂性.

SiriKit是Apple的工具包,用于通过Siri公开你app’s的功能.它是在iOS 10中引入的,尽管Siri在所有Apple平台上无处不在,但在撰写本文时,SiriKit仅适用于iOS和watchOS.当有人对Siri说话时,Siri将语音转换为文本,将文本转换为用户想要做的事情(发掘他们的Intent’意图’),从而导致行动并产生回应.发掘用户的Intent是SiriKit的核心.

用户如何通过Siri获App服务


SiriKit定义了用户可以进行的称为意图(Intent)的请求类型.相关意图被分组到域(Domain)中,以明确你可能在应用中支持哪些意图.例如 消息域具有发送消息,搜索消息以及将消息标记为已读或未读的意图.
你的应用扩展程序很少直接与用户通信.Siri通常会处理与用户的所有通信,并仅在需要你提供信息时才会呼叫你的App.你可以提供Intents UI应用程序扩展以自定义Siri和Maps显示的信息,这是可选的.

理解Intent(意图)

Siri的功能有限,仅在已知的Domains内工作.一个Domain是Siri知道的事物类别,比如拨打VoIP电话,发送短信,付款,工作列表和笔记,帮助你锻炼.每个Domain都有一系列Intents,这是Siri可以执行的操作.例如,在Payments Domain内,有发送付款,转账和支付账单这些Intent.在Ride booking Domain中,有请求乘车并获得乘车状态的Intent.在考虑采用SiriKit时,请查看Domain及其Intent,了解哪些内容对你的app有意义.

Intent需要有参数:发送消息时,向谁发送消息;什么时候预订,什么餐厅和多少人.因此,实现Intent可以执行三个步骤:

  1. 解析参数:帮助Siri理解用户提供的内容,包括向用户询问澄清,确认或其他信息的能力.
  2. 确认所有事情:最后确保所有Intent参数都是应有的,并且你的app可以处理用户的请求(e.g. 服务是在线状态且可用).
  3. 处理它

在这三个阶段中,你做什么和如何做取决于具体的Intent;本系列的第2部分将深入研究这三个阶段.但首先要做的是:让我们为app添加一个Intents Extension(意图扩展).

添加Intent扩展

由于与Siri的交互发生在你的App之外,因此App的SiriKit功能的实现为App扩展 - Intents App Extension.让我们创造一个!

在Xcode中启用Siri App功能

  1. 在Xcode中打开项目.
  2. 选择app target(应用目标)和Capabilities tab(功能选项卡).
  3. 启用Siri capabilities(功能).

这会将Siri Service添加到你的App ID中,以及Siri entitlement添加到项目中.

向Xcode项目添加Intents Extension Target

接下来,在项目中创建一个Intents Extension Target.

  1. 在Xcode中打开项目.
  2. 从菜单栏中选择:“File”>“New”>“Target”
  3. 选择“Intents Extension”.请注意,它不是“Siri Extension”,并且不需要“Intents UI Extension”,因为它是相关但不同的东西,然后单击Next.
  4. 选择新目标的选项,然后单击“完成”.

编辑Intents Extension的Info.plist

为了让系统知道应用程序可以处理的Intent,请编辑扩展的Info.plist.

  1. 打开Intents Extension的Info.plist
  2. 公开 NSExtension item.如果Info.plist中不包含它,请添加一个同名的字典类型.
  3. 公开 NSExtensionAttributes item.如果没有,请添加一个同名的字典类型.
  4. 编辑 IntentsSupported 扩展属性,如果需要,添加一个类型数组.每个条目都应该是你支持的Intent的类名字符串 - 每个支持的Intent都有一个条目.例如,如果你支持INSendMessageIntent,则应该有一个“INSendMessageIntent”条目.
  5. 如果你有任何需要解锁设备的Intent,可以在 IntentsRestrictedWhileLocked 扩展属性下的Intent列出 .
  6. NSExtensionPointIdentifier 扩展属性值应该为“com.apple.intents-service”.
  7. NSExtensionPrincipalClass 扩展属性应该通过项目信息给予正确的值.关于此的更多信息将在稍后讨论.

Build&Run

你的App现在具有与Siri一起使用的基本要素.我们来试试吧!是的,你可以在iOS模拟器中使用Siri.

  1. 确保Intents Extension target/scheme 选中.
  2. Run.
  3. 当Xcode提示运行app时,选择Siri.

运行Intents Extension,选择Siri作为要运行的应用程序.

当Siri波形界面出现时,你可以开始与Siri进行对话.说:“使用My Great App发送消息”,它就会调用你的Intents扩展.如果在IntentHandler.swift文件中设置断点,则可以通过解析,确认和处理阶段来查看扩展.请注意,根据你对Siri所说的内容,某些参数可能会被多次解析.

当然,作为开发人员,我们在开发过程中会一次又一次地做同样的事情.如果每次运行Intents Extension,你必须选择要运行的Siri app并说出你的短语,它可能会变得无聊.值得庆幸的是,Xcode提供了Siri Intent Query功能,:

  1. 编辑你的Intents Extension的scheme.
  2. 选择“Run” Panel.
  3. 选择“Info” tab.

如果你输入“Siri Intent Query”的阶段,Xcode将自动运行Siri并使用你的文本作为调用短语.如果将此字段留空,Xcode将在运行时提示你.

恭喜!你已成功将Intent扩展添加到app中,并开始使用Siri扩展你的app功能.

可在此处找到此完整源代码(以及整个SiriKit系列).

接下来

当然,以上对你的应用程序没有任何实用性或吸引力.在第2部分中,我们将做一些有用的事情并探索与Siri合作的三个关键概念:解析,确认和处理.

Progressive Web Apps on iOS

发表于 2018-07-26 | 分类于 iOS

伴随着iOS 11.3,Apple悄悄的增加了对”Progressive Web Apps(渐进式网络应用程序)”(PWAs) 新技术的基本支持.现在是时候看看它们是如何工作的,它们的能力和挑战是什么,以及如果您已经有已发布的PWA,您需要知道什么.

这个应用程序是一个PWA,它看起来全屏-具有离线能力,显示在iPad上.它也可以作为App Store中的任何其他原生应用程序出现在iPad Dock中.

如果你不知道什么是PWAs,它没有明确的定义.但它是一个使用Web技术创建的应用程序 - 无需打包或签名 - 可以离线工作,并且可以选择安装在操作系统中,它的外观和行为与任何其他应用程序一样.

大多数平台都没有涉及App Store流程 - 只有Edge / Windows 10正在强制PWA进入商店.

所以你是对的,你现在可以在没有App Store审核下的iOS上安装应用程序.这可能是Apple根本没有提及这种新能力的原因之一; 他们可能不想混淆用户.Safari上的发行说明甚至没有提到这些技术.

上图中您能说出原生谷歌地图应用与PWA版本之间的区别吗？

不管怎样,Apple不是PWA的创造者吗？

我们在这里说实话; 虽然Google与Chrome团队一起创造了PWA这个词,但这个想法最初是通过在original iPhone OS上的Safari上提供的.在2007年,Steve Jobs在WWDC中宣布了“另外一件事”:如何为原生 iPhone开发apps,有一个惊喜,这就是web apps! 那时App Store并不在原生应用开发的路线图中,并且这一年的设备下无法在本机中使用native SDK.从Apple的角度来看,即使在今天,PWA只是“来自主屏幕下的web apps”,而这个icon也被称为WebClip.

11年前这个想法并没有得到太多的关注,苹果视乎也忘了在平台上更新,所以虽然它有10多年的历史,但这些技术仍然存在错误和不一致的问题.几年后,其他平台克隆了这个想法,包括诺基亚N9上的MeeGo浏览器和Android上的Chrome.

Chrome帮助改进了技术以提供更好的体验,主要是使用Service Workers和Web App Manifest规范.从今天的iOS 11.3开始(2018年3月30日),Apple正在支持Chrome,Firefox, Samsung Internet,UC浏览器和Opera(主要仅限Android).其他桌面浏览器支持Service Workers,但Web App Manifest支持正在进行中.

Steve Jobs在WWDC 07中使用第一部iPhone呈现了PWA(具有不同的名称)

等一下,所以apps是没有通过任何App Store质量测试,对吧？🤔

对,这是正确的.但是,app仅在Browser或者具有安全性和具有执行模型的Web Platform下运行.这意味着您可以“publish”未在商店中批准的apps,例如公司员工的internal app(内部app) (是的,也包括成人内容),但您无法访问本机原始功能,例如iPhone X上的Face ID 或ARKit用于增强现实.或者至少,您需要等待Web Platform支持这些新功能.

PWA能在Safari内运行任意website,也可以在独立模式下运行,就像系统中的任何其他apps一样.如果您想知道PWAs是否正在使用Web视图,通过Safari或已安装的图标就可以确定是不是这种情况,但在其他浏览器中浏览时会出现这种情况或使用其应用程序内的浏览器,比如在Facebook中。

iOS上的PWAs能力

使用iOS上的Web Platform,您可以访问:

  • 地理位置
  • 传感器(磁强计,加速度计,陀螺仪)
  • 相机
  • 音频输出
  • 语音合成(仅连接耳机)
  • Apple Pay
  • WebAssembly,WebRTC,WebGL以及许多其他实验性功能.

你能在iPhone上检测出这些图标都是PWAs吗？

PWAs与原生iOS应用程序相比的局限性

  • 应用程序(app)可以存储最多50 Mb的离线数据和文件
  • 如果用户几周没有使用该应用程序(app),iOS将释放应用程序的文件.主屏幕上的图标仍然存在,访问时,应用程序将再次下载
  • 无法访问某些功能,例如Bluetooth,serial, Beacons,Touch ID,Face ID,ARKit,高度计传感器,电池信息
  • 无法在后台访问执行代码
  • 无法访问私人信息(联系人,location),也无法访问native social apps
  • 无法访问In App Payments和许多其他基于Apple的服务
  • 在iPad上,无法使用Side或Split Views与其他apps共享屏幕,PWAs将始终占据整个屏幕
  • 没有Push Notifications,没有 Icon Badge或Siri集成

即使您安装了带有图标且名称为Tinder的PWA,Siri也找不到它

哪些PWAs可以在Android上执行而不能在iOS上执行

  • 在Android上,您可以存储超过50 Mb
  • 如果您不使用该应用程序,Android不会删除文件,但它可以在有存储压力下删除文件.此外,如果用户安装或使用很多,PWA可以请求持久存储
  • BLE设备的蓝牙访问
  • 用于访问本机共享对话框的Web共享
  • 语音识别
  • 后台同步和Web Push Notifications
  • Web App Banner 邀请用户安装应用程序
  • 您可以自定义启动画面和所需的方向
  • 使用WebAPK和Chrome,用户无法安装多个PWA实例
  • 使用WebAPK和Chrome,PWAs会显示在“设置”下,您可以看到数据使用情况; 但在iOS上这一切都出现在Safari下
  • 使用WebAPK和Chrome,PWA管理其URL的地址,因此如果您获得PWA的链接,它将以独立模式打开,而不是在浏览器窗口中打开.

哪些PWAs可以在iOS上执行而不能在Android上执行

  • 用户可以在安装之前更改图标的名称
  • 它们可以在配置文件中配置,因此企业用户可以从公司接收PWA快捷方式.Safari使用术语WebClip来实现此功能;但它似乎没有阅读Web App Manifest(规范文档)

Configuration Profiles可以 include WebClip 或 PWA icons

但如果没有App Store,如何安装PWA？⚠️

这是iOS上最重大的挑战之一,因为没有来自Safari的提示或邀请(在Android上称为Web App Banners).因此,用户必须以某种方式在Safari中转到您的PWA URL,然后手动按“Share”图标,然后“Add to Home Screen”.没有征兆的表明您正在访问的网站来自PWA.

当您访问Tinder.com时,您可以忽略顶部的native app横幅安装提示,然后单击“Share”,“Add to Home Screen”.如果您要提供用户安装说明,请记住该按钮是根据系统语言进行本地化的

此外,App Store中提供的其他浏览器(如Chrome,Firefox,Brave或Edge)将无法安装PWA或使用Service Workers.

一旦安装,它将看起来像主屏幕上的任何其他图标.但它没有3D Touch菜单.如果再次安装相同的PWA,则会有另一个图标指向同一个PWA(幸运的是,已安装的文件将被共享).

我已经有了PWA,它会立即在iOS上运行吗？

用户更新到iOS 11.3后,您的PWA即可立即安装.您无需为iOS选择加入.每个PWA都可以安装.但这并不意味着一切都会按预期运作.

优步PWA看起来非常nice.但是,如果您单击“Login”或“Continue”,授权将进入Safari,因此您不在独立的PWA中…因此,它没用.

什么行不通

  • fullscreen(最大化)和minimal-ui (最小化)不会在iOS上工作,fullscreen需要独立触发,minimal-ui只是Safari的快捷方式.你可以使用 cover-fit viewport 扩展或已弃用的meta tag来获得相似的全屏(状态栏将存在,但在您的app上).
  • 如果您依赖后台同步,则应该有一个备份实现
  • 无法锁定PWA的方向
  • 状态栏上的主题颜色( theme-color)样式将不起作用; 您可以使用已弃用的meta tag用于黑色或白色状态栏,也可以使用CSS / HTML技巧来模拟主题颜色(theme-color).

  • 您的Android图标在iOS上可能看起来很糟糕,因为Apple不支持透明图标.

  • 此外,iOS不是从Web App Manifest中获取icons,而是从apple-touch-icon链接中获取图标.如果您未提供链接标记,则会将屏幕截图用于icons(参阅Google Keep PWA示例)
  • 没有启动画面,因此Manifest中的大多数颜色属性都会被忽略
  • 不会触发任何Manifest事件,因此如果您通过这些渠道跟踪安装,它将无法在iOS上运行(但您可以检查navigator.standalone).

  • 您的PWA不会在会话之间保持状态,如果用户退出PWA,它将在返回时重新启动,因此如果您需要用户验证电子邮件,SMS或执行two-factor身份验证,考虑到这一点,需要提供适当的解决方案.

  • 要在iOS上调试Service Workers,您需要安装Safari技术预览版或Safari 11.1,以及macOS 10.11.5,10.12.6和10.13.4更新.

Getting Started

Jump Right In

开始开发iOS应用程序(Swift)是学习创建在iPhone和iPad上运行的应用程序的完美起点. 下面的每课包含一个教程和完成它所需的概念信息. 这些课程相互构建,引导逐步创建简单真实世界的iOS应用程序.在完成课程并构建应用程序的过程中,将了解iOS应用程序开发中的概念,深入了解Swift编程语言,并熟悉Apple集成开发环境Xcode的许多有价值的功能(IDE).

Prerequisites/先决条件

在这些课程中,假定熟悉Swift programming language. 不需要成为 Swift master来完成课程,但如果能够舒适地阅读和理解Swift代码,你将会得到更多的收获.如果对Swift不太熟悉,在Swift Playgrounds应用程序中完成Learn to Code练习. 或者,可以通过The Swift Programming Language (Swift 4.1).的A Swift Tour. 两者都提供了Swift编程语言的坚实基础.

MVVM in Swift

引言

MVVM是Model-View-ViewModel的缩写,听起来很复杂,但它本质是MVC模式的改进版.MVVM 就是将其中View 的状态和行为抽象化,让我们将视图 UI 和业务逻辑分开.当然这些事 ViewModel 已经帮我们做了,它可以取出 Model 的数据同时帮忙处理 View 中由于需要展示内容而涉及的业务逻辑.

一般来说,MVVM通常与Functional Reactive Programming(FRP)结合使用,并且有很多FRP库,例如RxSwift和ReactiveCocoa.但是,如果有人不熟悉FRP,那么怎么在项目中使用MVVM呢.在本文中,将演示在没有FRP库的情况下使用MVVM.

Model View Controller

先开始使用标准的MVC方式.例子是在UITableView中显示一个排序后的integers Array(整数数组).另外,通过点击右上角的添加按钮,可以按照正确的顺序插入一个新的整数,并通过操作cell来删除一个integer.

在这里,先创建一个名为DemoViewController的UITableViewController子类,并在其中实现必要的UITableViewDataSource方法.另外,将插入逻辑放在addNewInteger方法中,将删除逻辑放在tableView(_tableView:,commit editingStyle:,forRowAt indexPath:)方法中.

Model View View Model

转换为MVVM的第一步是将其中View 的状态和行为抽象化.在我们例子中为一个排序的Integers table View,因此我们对table View的状态和行为进行处理,创建一个名为State的 struct,它存储table View的状态和行为.
首先在插入或删除一个integer后应该更新表视图,所以先编写一个名为EditingStyle的 emun,它包含insert,delete和none情况.
此外在State中创建一个editingStyle属性,并使用editingStyle属性上的Swift属性观察器功能(Swift’s property observers)更新sortedIntegers属性.

现在,根据这个思路,我们很容易创建View Model类,它包含插入和删除逻辑.此外,我们的View Controller只能通过我们的View Model访问状态信息.如果状态信息发生变化,我们的View Controller也可以通过View Controller的回调闭包来更新其表视图.

最后,修改视图控制器内部的代码以将所有内容都绑定起来. 创建一个viewModel属性并在viewDidLoad方法中实例化它.

总结

在项目中采用MVVM有几个优点.
首先,它使代码库更易于测试.Apple 的View Controller总是不方便进行集成测试,采用MVVM模式可以将大部分业务逻辑迁移到视图模型中为业务逻辑编写测试提供了便利性和可能性.
其次,遵循这种模式可以使代码库更加一致并带来更多的可读性.
此外,可以进一步降低与FRP库中的UI binding(UI 绑定)和Asynchronous Chaining(异步链接)的复杂性.

Swift是编写软件的绝妙方式,无论是用于手机,台式机,服务器还是其他运行代码的软件. 这是一种安全,快速和交互式的编程语言,融合了现代语言思维的最佳水平和更广泛的Apple工程文化的智慧以及开源社区的多元贡献. 编译器针对性能进行了优化,并且语言针对开发进行了优化,而无需牺牲任何一种.

Swift代码被编译和优化,以充分利用现代硬件.语法和标准库的设计基于指导原则,写代码的显而易见的方式也应该表现得最好. 它的安全性和速度相结合使Swift成为从”Hello,world！”到整个操作系统的绝佳选择.

Control Flow 控制流

使用if和switch来创建条件,并使用for-in,while和repeat-while来创建循环. 条件或循环变量的括号是可选的. 需要身体周围的大括号.

在if语句中,条件必须是Boolean表达式 - 这意味着如if score {…}这样的代码是一个错误,而不是隐式的比较为zero.

使用if和let可能会丢失的值. 这些值表示为可选项. 可选值包含一个值或包含nil以指示缺少一个值. 在值的类型后面写一个问号(?),将该值标记为可选.

练习: 把optionalName改成nil,greeting会是什么？添加一个else语句,当optionalName是nil时给greeting赋一个不同的值.

Swift是iOS,macOS,watchOS和tvOS应用程序开发的新编程语言. 如果有 C 或者 Objective-C 开发经验的话,会发现 Swift 的很多内容都是熟悉的.

Swift 包含了 C 和 Objective-C 上所有基础数据类型,Int表示整型值;Double和Float表示浮点型值;Bool是布尔型值;String是文本型数据.Swift 还提供了三个基本的集合类型,Array,Set和Dictionary,详见集合类型.

就像 C 语言一样,Swift 使用变量来进行存储并通过变量名来关联值.在 Swift 中,广泛的使用着值不可变的变量,它们就是常量,而且比 C 语言的常量更强大.在 Swift 中,如果你要处理的值不需要改变,那使用常量可以让你的代码更加安全并且更清晰地表达你的意图.

除了熟悉的类型,Swift 还增加了 Objective-C 中没有的高阶数据类型比如元组(Tuple).元组可以让你创建或者传递一组数据,比如作为函数的返回值时,你可以用一个元组可以返回多个值.

Swift还引入了可选(Optional) 类型,它处理value缺失的情况. Optional表示,”there is a value, and it equals x“或”there isn’t a value at all”. 使用Optional类似于在Objective-C中使用nil,但它可以适用于任何类型,不仅仅是类. 可选类型比Objective-C中的nil指针更安全,更具表现力.它是Swift许多强大特性的重要组成部分.

Swift 是一门类型安全的语言,可选类型就是一个很好的例子.Swift 可以让你清楚地知道值的类型.如果你的代码期望得到一个String,类型安全会阻止你不小心传入一个Int.你可以在开发阶段尽早发现并修正错误.

Type Safety and Type Inference/类型安全和类型推断

Swift 是一个类型安全(type safe)的语言.类型安全的语言可以让你清楚地知道代码要处理的值的类型.如果你的代码需要一个String,你绝对不可能不小心传进去一个Int.

由于 Swift 是类型安全的,所以它会在编译你的代码时进行类型检查(type checks),并把不匹配的类型标记为错误.这可以让你在开发的时候尽早发现并修复错误.

当你要处理不同类型的值时,类型检查可以帮你避免错误.然而,这并不是说你每次声明常量和变量的时候都需要显式指定类型.如果你没有显式指定类型,Swift 会使用类型推断(type inference)来选择合适的类型.有了类型推断,编译器可以在编译代码的时候自动推断出表达式的类型.原理很简单,只要检查你赋的值即可.

Assertions and Preconditions /断言和前置条件

断言和前置条件是在运行时发生的检查. 在执行任何进一步的代码之前,您可以使用它们来确保满足基本条件.如果断言或前提条件中的布尔条件评估为true,则代码继续照常执行. 如果条件评估为false,则该程序的当前状态无效;代码执行结束,并且您的应用程序被终止.

使用断言和前置条件来表达所做的假设和编码时的期望,以便可以将它们包含在代码中. 断言有助于在开发过程中发现错误和不正确的假设,并且前置条件可帮助您检测生产中的问题.

除了在运行时验证您的期望之外,断言和前置条件也成为代码中有用的文档形式. 与上述错误处理中讨论的Error Handling 不同,断言和先决条件不用于可恢复或预期的错误. 由于失败的断言或前置条件表示无效的程序状态,因此无法捕获失败的断言.

使用断言和前置条件不能取代设计代码的方式,使得不可能出现无效条件. 但是,使用它们来强制执行有效的数据和状态会导致应用程序在发生无效状态时可预测终止,并且有助于使问题易于调试. 一旦检测到无效状态,立即停止执行也有助于限制由该无效状态引起的损害.

断言和前置条件之间的区别在于它们被检查时:断言仅在调试版本中检查,但在调试版本和生产版本中都会检查了前置条件. 在生产构建中,断言内的条件不被评估. 这意味着您可以在开发过程中使用尽可能多的断言,而不会影响生产性能.

Comparing Classes and Structures 类和结构体对比

Swift 中类和结构体有很多共同点.共同处在于:

  • 定义方法用于提供功能

  • 定义构造器用于生成初始化值
  • 通过扩展以增加默认实现的功能

Choosing Between Classes and Structures/类和结构体的选择

在代码中,可以使用class和struct来定义你的自定义数据类型.

然而,struct实例总是通过value传递,class实例总是通过reference传递.这意味两者适用不同的任务.当在考虑一个工程项目的数据结构和功能的时候,需要决定每个数据结构是定义成class还是struct.

按照通用的准则,当符合一条或多条以下条件时,考虑构建结构体:

  • 该数据结构的主要目的是用来封装少量相关简单数据值.
  • 有理由预计该数据结构的实例在被赋值或传递时,封装的数据将会被拷贝而不是被引用.
  • 该数据结构中储存的值类型属性,也应该被拷贝,而不是被引用.
  • 该数据结构不需要去继承另一个既有类型的属性或者行为.

举例来说,以下情境中适合使用结构体:

  • 几何形状的大小,封装一个width属性和height属性,两者均为Double类型.
  • 一定范围内的路径,封装一个start属性和length属性,两者均为Int类型.
  • 三维坐标系内一点,封装x,y和z属性,三者均为Double类型.

在所有其它案例中,定义一个类,生成一个它的实例,并通过引用来管理和传递.实际中,这意味着绝大部分的自定义数据构造都应该是类,而非结构体.

Assignment and Copy Behavior for Strings, Arrays, and Dictionaries/字符串、数组、和字典类型的赋值与复制行为

Swift 中,许多基本类型,诸如String,Array和Dictionary类型均以struct的形式实现.这意味着被赋值给新的常量或变量,或者被传入函数或方法中时,它们的值会被拷贝.
Objective-C 中NSString,NSArray和NSDictionary类型均以class的形式实现.它们在被赋值或者被传入函数或方法时,不会发生值拷贝,而是传递现有实例的引用.

NOTE

以上是对字符串、数组、字典的“拷贝”行为的描述.在代码中,拷贝行为看起来似乎总会发生.然而,Swift 在幕后只在绝对必要时才执行实际的拷贝.Swift 管理所有的值拷贝以确保性能最优化,所以也没必要去回避赋值来保证性能最优化.

Property Observers/属性观察者

属性观察者会观察并对属性值的变化做出回应.每当一个属性的值被设置时,属性观察者都会被调用,即使这个值与该属性当前的值相同.

可以为定义的任意存储属性添加属性观察者,除了lazy存储属性,也可以通过在子类里重写属性来为任何继承属性(无论是存储属性还是计算属性)添加属性观察者.属性重载将会在重写中详细描述.

可以选择将这些观察者或其中之一定义在属性上:

  • willSet 会在该值被存储之ff前被调用.
  • didSet 会在一个新值被存储后被调用.

如果实现了一个 willSet 观察者,新的属性值会以常量形式参数传递.可以在willSet实现中为这个参数定义名字.如果没有为它命名,那么它会使用默认的名字 newValue .

同样,如果实现了一个didSet观察者,一个包含旧属性值的常量形式参数将会被传递.可以为它命名,也可以使用默认的形式参数名oldValue .如果在属性自己的 didSet 观察者里给自己赋值,赋值的新值就会取代刚刚设置的值.

didSet 观察者在 totalSteps 的值更新后调用.它用旧值对比 totalSteps 的新值.如果总步数增加了,就打印一条信息来表示接收了多少新的步数. didSet 观察者不会提供自定义的形式参数名给旧值,而是使用 oldValue 这个默认的名字.

Global and Local Variables/全局和局部变量

上边描述的计算属性和观察属性的能力同样对全局变量和局部变量有效.

全局变量是定义在任何函数、方法、闭包或者类型环境之外的变量.
局部变量是定义在函数、方法或者闭包环境之中的变量.

在之前章节中所遇到的全局和局部变量都是存储变量.存储变量,类似于存储属性,为特定类型的值提供存储并且允许这个值被设置和取回.

总之,同样可以定义计算属性以及给存储变量定义观察者,无论是全局还是局部环境.计算变量计算而不是存储值,并且与计算属性的写法一致.

NFC结构图

Type 1 Tag

Type 1 Tag 是最简单的产品.也是最慢的芯片,由于它提供的简单性,也可以在芯片上填充更多的内存.这种标签很简单,往往是最便宜的,但可能缺乏某些应用程序所需的功能.
通常使用Type 1 Tag用于以下场景中的应用:

  • One-time provisioning
  • Read-only applications
  • Business cards
  • Pairing devices with Bluetooth
  • Reading a specific tag when more than one tag is present

Type 2 Tag

Type 2 Tag 最流行的产品,因为它相对便宜的价格以及提供足够的功能,以满足广泛的需求. Type 2 标签也比Type 1标签快,所以可以依靠它来完成用户之间几乎即时通信的应用.
通常使用Type 2 Tag用于以下场景中的应用:

  • Low-value transactions
  • Day transit passes
  • Event tickets
  • URL redirects

Type 3 Tag

Type 3 Tag 在与此组中其他标签不同的标准.索尼FeliCa Tag 是日本的创新,在亚洲广泛使用,主要在日本使用.这是一个复杂的标签,提供了广泛的功能,但也带有相对较高的价格标签.
通常使用Type 3 Tag用于以下场景中的应用:

  • Transit tickets
  • E-money
  • Electronic ID
  • Membership cards
  • E-tickets
  • Health care devices
  • Home electronics

Type 4 Tag

Type 4 Tag 为所有标签提供了最大的灵活性和内存.它的价格中等偏高,也取决标签的内存量.这个标签最重要的原因是其安全性,它提供了执行认证所需的功能.另外这是唯一支持ISO 7816安全性的标签,并允许NDEF内容的自我修改.
鉴于此标签提供的额外功能,通常会看到它用于

  • transit ticket applications

Type 5 Tag

Type 5 Tag 支持ISO / IEC 15693规范.
在这种情况下,NFC Forum 选择支持主动通信模式,这样可以实现与NFC Forum 已经支持的 RF technologies 类似的整体数据传输性能.读取距离(reading distance)与其他NFC技术完全相同.
由于标准要求支持此模式,因此支持Type 5标签的支持NFC的设备可以读取ISO / IEC 15693标签.
通常使用Type 5 Tag用于以下场景中的应用:

  • Library books, products, and packaging
  • Ticketing (such as ski passes)
  • Healthcare (medication packaging)

iOS 11使用NFC前需要注意一下几点

  • 需要开启一个session,与其他session类似,同时只能开启一个
  • 需要App完全在前台模式
  • 每个session最多扫描60s,超时需再次开启新session
  • 配置读取单个或多个Tag,配置为单个时,会在读取到第一个Tag时自动结束session
  • 隐私描述会在扫描页面显示

Core NFC

发表于 2017-12-29 | 分类于 iOS

官方文档链接

用于检测NFC(近场通讯) 标签和读取包含NDEF数据的信息.

Overview

你的app可以通过NFC读取标签,从而给予用户提供更多有关其物理环境和真实世界的对象内容信息.比如:你的app可能可以给用户提供商店里的商品信息或者是他们在博物馆参观时的陈列品的信息.

要求

  • SDK要求iOS 11.0+
  • 目前只有iPhone 7 and iPhone 7 Plus支持读取NFC NDEF标签
  • Supports ISO14443 and ISO15693 tags.

使用Core NFC,你可以读取到包含NDEF (NFC数据交换格式)格式数据的1到5类型的NFC 标签.你的app可以通过创建一个NFC NDEF reader session对象同时提供一个delegate代理来读取标签.当一个正在运行的reader session对象发现了一个包含NDEF信息的标签后,就会将这个信息传递给它的delegate代理.这个delegate代理除了能读取消息还能处理session变为无效的情况.
要启用您的应用程序来检测NFC标签,请打开Xcode项目中的Near Field Communication Tag阅读功能. 要了解有关向项目添加功能的更多信息,see Add a capability to a target.

Core NFC包含以下内容

  • Reader Sessions

    • class NFCNDEFReaderSession
      用于检测NFC NDEF标签的reader session类
    • protocol NFCReaderSessionProtocol
      定义了与reader session交互的通用接口
    • class NFCReaderSession
      用于描述一个用来检测NFC标签的reader session的抽象类

  • NFC Tags

    • protocol NFCTag
      定义了NFC或RFID(Radio Frequency Identification,射频识别)标签的属性
    • class NFCTagCommandConfiguration
      此类提供了一组的参数可以让你用来配置NFC标签命令

  • NDEF Messages

    • class NFCNDEFMessage
      NFC NDEF消息类,由一组payload记录组成
    • class NFCNDEFPayload
      此类表示在NFC NDEF消息里的一个payload记录
    • enum NFCTypeNameFormat
      NFC类型枚举,用于指定NFC NDEF消息中的内容类型.
      ​ NFCTypeNameFormatAbsoluteURI
      ​ The data uses the uniform resource identifier standard.
      ​ NFCTypeNameFormatEmpty
      ​ 有记录是空的,不包含任何信息
      ​ NFCTypeNameFormatMedia
      ​ The data uses the media type defined by RFC 2046.
      ​ NFCTypeNameFormatNFCExternal
      ​ The data contains a value that uses the NFC record type definition for external types.
      ​ NFCTypeNameFormatNFCWellKnown
      ​ The data contains a value that uses a type that follows the NFC record type definition.
      ​ NFCTypeNameFormatUnchanged
      ​ The data is not the first record in a series of records containing chunked data.
      ​ NFCTypeNameFormatUnknown
      ​ The data type is unknown.

  • Errors

    • enum NFCReaderError
      reader sessions和tags错误类型标识
      ​ NFCReaderSessionInvalidationErrorFirstNDEFTagRead
      ​ The first NDEF tag read by this session is invalid.
      ​ NFCReaderSessionInvalidationErrorSessionTerminatedUnexpectedly
      The reader session terminated unexpectedly.
      ​ NFCReaderSessionInvalidationErrorSessionTimeout
      ​ The reader session timed out.
      ​ NFCReaderSessionInvalidationErrorSystemIsBusy
      The reader session failed because the system is busy.

      Connection to the tag has been lost.
      NFCReaderSessionInvalidationErrorUserCanceled = 200
      ​ The user canceled the reader session.
      NFCTagCommandConfigurationErrorInvalidParameters = 300
      ​ The tag has been configured with invalid parameters.
      NFCReaderTransceiveErrorRetryExceeded
      ​ Too many retries have occurred.
      NFCReaderTransceiveErrorTagResponseError
      ​ The tag has responded with an error.
      NFCReaderErrorUnsupportedFeature = 1
      The reader session does not support this feature.
      NFCReaderErrorSecurityViolation
      ​ A security violation associated with the reader session has occurred.

      The error domain for errors associated with Core NFC APIs.
      NSString *const NFCErrorDomain;
      ​ The error domain for errors associated with Core NFC APIs.

了解 NDEF-Records

为了使用NFC-tags,理解NDEF(NFC Data Exchange Format)规范是非常重要的.每当CoreNFC发现一个新标签,didDetectNDEFs委托方法将提供一个NDEF messages（[NFCNDEFMessage]）.
通常,只包含一个NDEF message,但是规范保持灵活性,以便同时提供多个消息.
每个NFCNDEFMessage都包含一个payload-records([NFCNDEFPayload]),用于存放开发人员感兴趣的实际信息.目前,CoreNFC框架中有四个属性用于访问这些属性:

  1. typeNameFormat指定payload 是哪种NDEF 类型. 详见NFCTypeNameFormat NFC类型枚举,用于指定NFC NDEF记录中的数据类型.
  2. typepayload的类型.
  3. identifierpayload的唯一标识符.
  4. payloadpayload的数据,数据格式取决于 typeNameFormat 指定的类型.

用户体验

  1. 扫描 NDEF-tag 通常会在重新启动iPhone后立即执行一次.从那时起,它可能起作用或不起作用,通常不起作用,需要重新启动iPhone.在iOS 11下是出现过的.
  2. 如果扫描 RFID-tag 是最新的或者不包含 NDEF-tag(e.g. a credit-card),reader将会times out(error 201).
  3. 如果扫描 RFID-tag 包含加密分区,reader session将会throws error 200(readerSessionInvalidationErrorUserCanceled)

RFID功能 ⚠️ 警告 目前iOS11还不支持

我们使用NFCNDEFReaderSession来处理 NDEF NFC-Chips.在CoreNFC中实际还有另一个类,叫做NFCISO15693ReaderSession用来处理RFID-tags,
其规范为ISO15693,可以通过公布的delegates 和 NFCISO15693Tag class来进行处理.
你甚至可以发送自定义命令到 RFID-chip上,通过readerSession:didDetectTags: delegate 和NFCISO15693CustomCommandConfiguration class 来实现.
不过目前 iOS 11 Beta 1-2会抛出一个功能不支持的错误,Beta 3-4将打开扫描对话框,但立即出现一个abort的错误.
这可能会导致com.apple.developer.nfc.readersession.formats授权密钥内的另一个值.
使用ISO15693或RFID之类的东西到目前为止都不能工作,并且不能完成构建.我们看看未来Apple 是否在iOS 11 + Beta 更新此特性 🙂
